The 5 Mechanics to Encode Once

Syntax is arbitrary. Encode each one through doing, not reading. One rep each from blank IDE.

1 Index vs Value
Invariant: index = WHERE it is. value = WHAT it is. nums[index] = WHAT at WHERE. Never confuse the two in a variable name.
# index: position in array
# value: the actual element
nums = [2, 7, 11, 15]
i = 0           # i is an index
val = nums[i]  # val is a value

# for loops give you value directly
for num in nums:        # num = value
    pass

for i, num in enumerate(nums):  # both
    pass
2 Variable Types — Always Ask TYPE First
Invariant: Before naming any variable, ask: what TYPE is this? A dict is not a counter. A list is not a string.
seen = {}            # dict  — maps key → value
count = 0           # int   — accumulates a number
result = []         # list  — collects items
max_len = 0        # int   — tracks a running max
left = 0            # int   — a pointer (index)

# defaultdict auto-initializes missing keys
from collections import defaultdict
groups = defaultdict(list)  # missing key → []
3 Inside Loop vs Outside Loop
Invariant: Inside loop = resets every iteration (fresh). Outside loop = persists across iterations (accumulates). Wrong placement = wrong answer.
# WRONG — curr_sum resets every element
for num in nums:
    curr_sum = 0       # ← inside: always 0
    curr_sum += num

# CORRECT — curr_sum accumulates
curr_sum = 0        # ← outside: persists
for num in nums:
    curr_sum += num
4 New Purpose = New Variable Name
Invariant: The moment a variable's purpose changes, make a new box with a new name. Reusing the same name for a different purpose is the root cause of 80% of rep 2 bugs.
# sliding window — two DIFFERENT indices, two names
left = 0                 # left boundary pointer
for right in range(len(s)):  # right boundary pointer
    pass

# hashmap — key stores index, but it IS an index not a value
seen[s[right]] = right  # value = right (index)
# reading it back: seen[char] gives the index of char
5 Loop Exit State — Ask What Each Variable Holds
Invariant: After any while loop terminates, stop and ask: what is each variable's value RIGHT NOW? The answer to the problem lives in one of them.
# linked list reversal — what holds the answer at exit?
curr = head
prev = None
while curr is not None:
    next_node = curr.next
    curr.next = prev
    prev = curr
    curr = next_node
# at exit: curr = None, next_node = None, prev = NEW HEAD
return prev  # always prev for linked list reversal

Sliding Window — LC3 Longest Substring Without Repeating locked
def length_of_longest_substring(s):
    seen = {}       # char → last seen index
    left = 0
    max_len = 0
    for right in range(len(s)):
        if s[right] in seen and seen[s[right]] >= left:
            left = seen[s[right]] + 1
        seen[s[right]] = right
        max_len = max(max_len, right - left + 1)
    return max_len

# REP 2 BUG: used `i` instead of `right`, then used `i` again
# as a separate loop counter — same name, different purpose.
# seen[i] = i means both key and value were the same variable.
# Fix: left/right are pointer names, seen[s[right]] = right is clear.
Two Pointers — Two Sum II pending rep
def two_sum(nums, target):
    left, right = 0, len(nums) - 1
    while left < right:
        curr = nums[left] + nums[right]
        if curr < target:
            left += 1
        elif curr > target:
            right -= 1
        else:
            return [left, right]
    return []

# REP 2 BUG: named sum variable `sum` — shadows Python built-in.
# Also used `left` as both the pointer AND the result index.
# Fix: curr = current sum (new purpose = new name).
The Transfer Layer — One Invariant Per Family

Once you have the anchor sentence you reconstruct from scratch, not from memory. This is where SICP kicks in.

HashMap
"store while scanning, retrieve in O(1)"
The invariant: You've already seen something. You store it. Later you check if what you need is already stored. The dict is the bridge between past and present in the same scan.
signal
grouping / counting / lookup / existence check
key
what you fingerprint on (sorted string, char, value)
value
what you retrieve (list of words, index, count)
# Transfer: any grouping problem
# fingerprint → key, item → value
# "".join(sorted(word)) is a fingerprint
# counts[char] += 1  is a count fingerprint
Two Pointers
"squeeze inward until they meet"
The invariant: Sorted array. Two ends. The comparison at any step tells you which side is wrong. You eliminate one side. You never need to check both simultaneously.
signal
sorted array + find pair / partition / palindrome
stop
left >= right
move
too small → left++, too big → right--, equal → answer
Sliding Window
"expand right until broken, shrink left until fixed"
The invariant: right always moves forward. left only moves to repair a violation. seen stores the last index of each char — not just whether it exists, but WHERE it was, so you can jump left past it.
signal
contiguous subarray / substring + constraint
seen value
last index (not bool) — needed to jump left correctly
guard
seen[char] >= left — only care if char is inside current window
Trees
"ask None, ask one node, combine children"
The invariant: You never need to see the whole tree. You ask one node one question. None always has a definite answer (0, -inf, True). The combine step is the entire problem.
signal
depth / value / relationship / path / count
base
None → what? (0 for depth, -inf for max, True for symmetric)
combine
max(left, right) + 1, or left and right, or left + right
Linked List
"save next, reverse pointer, slide forward"
The invariant: Three variables: prev, curr, next_node. Always in that order. At loop exit, curr = None. prev = answer. Never return curr.
signal
traverse / reverse / detect / merge
order
next_node = curr.next → curr.next = prev → prev = curr → curr = next_node
exit
curr is None, prev is new head
Rep 2 Full Mistake Log — Root Cause Analysis

Every bug traces to one of two root causes: variable name collision, or wrong loop placement.

linked list reversal — rep 2
Wrote: return next_node → then return reverse_list → then return curr
Why: loop logic was perfect, blanked on exit state of each variable
Fix: at loop exit curr=None, next_node=None, prev=new head. Answer is always in prev.
find_max — rep 2
Wrote: left = node.left / right = node.right — stored node objects not values
Why: confused the node object with the result of recursing into it
Fix: node.left = child object. max_value(node.left) = max value IN that subtree. Always recurse to get results.
group_anagrams — rep 2
Wrote: used result as both the dict name AND the loop variable
Why: new purpose, same name. dict got shadowed by the string from the loop.
Fix: new purpose = new box name. Loop var = word. Dict = result (or groups).
sliding window — rep 2
Wrote: used i for both the pointer and as a loop index in a different context
Why: collision between loop variable names across mental contexts
Fix: left/right are semantic names. i is anonymous. Prefer semantic over single-letter for pointers.
two_sum — rep 2
Wrote: sum = nums[left] + nums[right] — shadows Python built-in
Why: sum is a real Python function. Also used left as both pointer and result index.
Fix: curr = current sum. Never name variables after built-ins (sum, list, map, min, max).
MinStack — rep 1
Wrote: return stack[-1] and min_stack[-1] in top(). Also called self.min_stack.top()
Why: Python lists don't have .min() or .top() — those are Java Stack methods. Also and returns last truthy value, not a tuple.
Fix: return self.stack[-1] for top(). return self.min_stack[-1] for get_min(). Use [-1] not .top().
Global root cause: Every bug above traces to one of two patterns.
(1) Variable name collision — same name, different purpose, in the same scope.
(2) Wrong exit / wrong variable — not stopping to ask "what does each variable hold right now?"

MinStack — Full Implementation locked
class MinStack:
    def __init__(self):
        self.stack = []
        self.min_stack = []     # parallel stack tracking minimums

    def push(self, val):
        self.stack.append(val)
        min_val = min(val, self.min_stack[-1] if self.min_stack else val)
        self.min_stack.append(min_val)

    def pop(self):
        self.stack.pop()
        self.min_stack.pop()     # always pop both together

    def top(self):
        return self.stack[-1]    # [-1] not .top() — Python list

    def get_min(self):
        return self.min_stack[-1]  # [-1] not .min()

# THREE BUGS FROM REP 1:
# 1. return stack[-1] and min_stack[-1]  → and returns last truthy value
# 2. self.min_stack.top()  → .top() is Java, not Python
# 3. return min(self.min_stack.top())  → double wrong: .top() + wrapping in min()
Class Mechanics — Python vs Java mental model
Key invariant: In Python, self is always explicit. Every method takes self as first parameter. Every attribute access is self.thing. There is no implicit this.
class Node:
    def __init__(self, value):  # only value parameter
        self.val = value
        self.left = None          # initialized to None, not passed in
        self.right = None

# WRONG (from test_recall.py rep 1):
# def __init__(self, value, left, right):
#     self.next = next   ← mixing linked list and tree attributes

# Python list as stack:
# push  →  stack.append(val)
# pop   →  stack.pop()
# peek  →  stack[-1]       ← NOT .top(), NOT .peek()
# empty →  len(stack) == 0 or not stack

def backtrack(state, choices):
